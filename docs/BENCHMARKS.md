# Benchmark Guide

Every quantitative claim in the paper is **measured at runtime** and written to
`results/*.json` with the host CPU, OS build, and toolchain versions embedded —
nothing is hard-coded. This guide explains what each benchmark measures, how to
run it, and where its output maps in the paper. The authoritative, fully
detailed procedure (including the §B measurement discipline) is
[`../REPRODUCE.md`](../REPRODUCE.md); the claim-to-file map is
[`TRACEABILITY.md`](TRACEABILITY.md).

> **Read this first.** Benchmark numbers are hardware- and thermal-state
> dependent. The committed `results/*.json` were measured on a 13th-Gen Intel
> Core i7-1355U (2 P + 8 E cores, 15 W ULV) under WSL2. **Your numbers will
> differ** — that is expected. The scientific claims (sub-second native proving,
> the ordering vs. the centralized baseline, the negative-test pass) are robust
> to the hardware; the exact milliseconds are not.

## The controlled campaign (recommended)

All published numbers come from the single orchestrated campaign, which runs
every measured artifact **strictly serially on a quiesced machine**:

```bash
npm run campaign      # = make bench
# circomspect gate -> pqc -> zk -> baseline -> gas -> negative
#   -> malicious-wallet probe -> e2e -> latency budget -> RESULTS.md
npm run tables        # regenerate RESULTS.md + the divergence table
```

### §B measurement discipline (enforced automatically)

1. **Quiesce guard** — every timing bench calls `assertQuiesced()`
   (`harness/bench_env.ts`) and **refuses to start** if build tooling, another
   prover, surplus Node processes, high Windows CPU load, or WSL `loadavg > 1.0`
   are present. Override only with `PQID_BENCH_FORCE=1` (recorded in the JSON).
2. **Recorded controls** — affinity policy + live probe, CPU governor, power
   source, filesystem basis, and the quiesce report are embedded in every
   results file under `controls`.
3. **≥3 independent invocations** per metric (separate processes), with inter-run
   median spread and coefficient of variation; CV > 10% is flagged `⚠ UNSTABLE`.
4. **Median headline** when within-run σ/mean > 0.2 (applies to snarkJS proving).
5. **Native filesystem** — rapidsnark inputs are staged on WSL-native ext4 and
   SHA-256-verified after copy; never the slow `/mnt/c` 9p mount.

## Smoke runs (never for published numbers)

Lower the iteration counts via environment variables for a quick check:

```bash
PQID_ZK_RUNS=1 PQID_ZK_SNARKJS_N=3 PQID_ZK_RAPIDSNARK_N=5 PQID_ZK_WARMUP=1 npm run bench:zk
```

The `$comment` field in `fixtures/bench-config.json` records the published
methodology (N=1000 for PQC/witness/snarkJS/verify, N=100 for rapidsnark, 5
warmup, revoked-set size 8). Overrides are for smoke testing only.

## Interpreting the output

- `RESULTS.md` is regenerated by `harness/generate_results.ts` from the JSON and
  is **not edited by hand**. It maps every measured cell to its paper table cell
  and auto-generates a **complete divergence table** (any |Δ| > 5% vs the paper,
  with curated cause text), mirrored into `MANUSCRIPT_RECONCILIATION.md`.
- Each results file records its `controls`, `host`, and (for WSL-side work) `wsl`
  metadata, so a number can always be traced to the machine that produced it.
- The thermal-state characterization in `RESULTS.md` explains why single-
  authentication (cool) and sustained (heat-soaked) figures differ on this ULV
  laptop; both are reported.
